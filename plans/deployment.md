# Store Deployment Plan

This document is a practical, step-by-step walkthrough for submitting TabLocker to the
Chrome Web Store, Microsoft Edge Add-ons, and Firefox Add-ons (AMO). It reflects how the
build actually works today (`build.js`) and should be updated if the build or manifest
structure changes.

## Background: what the build produces

`yarn build` (`node build.js`) runs once and produces two independent, store-ready
directories:

- `dist/chrome/` — built from the base `manifest.json` (Manifest V3, `background.service_worker`,
  `action.default_popup`, `permissions: ["storage", "tabs"]`). This manifest is also what Edge
  consumes, since Edge is Chromium-based and reads the same Manifest V3 shape.
- `dist/firefox/` — built from `manifest.json` merged (shallow spread) with
  `manifest.firefox.json`, which adds `browser_specific_settings.gecko` (extension ID
  `tablocker@arronei.github.io` and `data_collection_permissions`). Firefox requires this
  `gecko.id` field for signing/updates; Chrome and Edge ignore it.

Each `dist/<browser>/` directory contains:

- Bundled JS entry points: `background/index.js`, `history/index.js`, `popup/index.js`
  (esbuild, ESM format, with sourcemaps)
- Static assets copied verbatim: `popup/index.html`, `popup/style.css`,
  `history/index.html`, `history/style.css`
- `icons/` — copied from `public/icons` (excluding `.gitkeep`). **Note:** as of this
  writing `public/icons` only contains a `.gitkeep` placeholder — actual PNG icon files
  (e.g. 16/32/48/128px) need to be added to `public/icons` and referenced from
  `manifest.json`'s `icons` field before store submission, since all three stores require
  icon assets both in the package and in the listing.
- `manifest.json` — the merged, browser-specific manifest written directly into the output
  directory (not copied from source, but generated by `build.js`).

Each browser's output directory is fully self-contained and is what gets zipped for
submission — never zip the repo root or `src/`.

## 1. Pre-submission checklist

Run this before starting any store submission, and repeat it for every subsequent release:

1. **Confirm the version has been bumped.** This repo uses `release-please`
   (`.github/workflows/release-please.yml`) to open an automated "release PR" against
   `main` that bumps `version` in `package.json` (and, since `build.js` reads
   `manifest.json` independently, verify `manifest.json`'s own `"version"` field is kept in
   sync — `release-please` with `release-type: node` updates `package.json` but does not
   automatically touch `manifest.json`, so this needs to be checked/updated by hand as
   part of merging the release PR). Store listings key off the manifest version, not
   `package.json`.
2. **Run the full CI suite locally**, matching what `.github/workflows/ci.yml` runs on every
   push/PR:
   - `yarn install --frozen-lockfile`
   - `yarn lint` (ESLint + Stylelint)
   - `yarn typecheck` (`tsc --noEmit`)
   - `yarn test:unit:coverage` (or `yarn test:unit` for a quicker pass)
   - `yarn build`
   - optionally `yarn playwright install --with-deps chromium && yarn test:e2e` for
     the e2e suite (this also implicitly rebuilds `dist/chrome` via the `pretest:e2e` hook)
3. **Run the Firefox extension linter** against the built Firefox output:
   ```
   yarn build
   yarn lint:ext
   ```
   This runs `web-ext lint --source-dir dist/firefox` and will flag manifest/API issues
   that AMO's own automated validator would also catch — fixing these locally avoids a
   round-trip rejection from Mozilla's review pipeline.
4. **Manually smoke-test the unpacked build in each target browser** before packaging:
   - **Chrome**: `chrome://extensions` → enable Developer Mode → "Load unpacked" →
     select `dist/chrome`.
   - **Edge**: `edge://extensions` → enable Developer Mode → "Load unpacked" →
     select `dist/chrome` (Edge uses the same Chromium package, so no separate `dist/edge`
     build is needed).
   - **Firefox**: `about:debugging#/runtime/this-firefox` → "Load Temporary Add-on" →
     select `dist/firefox/manifest.json`. (Alternatively
     `yarn dlx web-ext run --source-dir dist/firefox` for a live-reloading dev session.)
   - In each browser, verify: the popup opens and locks a tab, the history page renders
     and shows tracked history, the extension icon/badge appears correctly, and there are
     no console errors in the service worker / background context.
5. Confirm `LICENSE`, `README.md`, and any privacy-policy text referenced from the store
   listings are current — all three stores ask for a privacy policy URL or statement given
   this extension uses the `tabs` permission.

Only proceed to store submission once all of the above pass cleanly.

## 2. Chrome Web Store

**Account setup (one-time):**

1. Sign in to the [Chrome Web Store Developer Dashboard](https://chrome.google.com/webstore/devconsole)
   with a Google account.
2. Pay the one-time $5 USD developer registration fee if this Google account has not
   registered as a Chrome Web Store developer before. This is a single lifetime fee per
   developer account, not per extension.

**Packaging:**

1. Run `yarn build` to produce a fresh `dist/chrome`.
2. Zip the _contents_ of `dist/chrome` (not the `dist` or `chrome` folder itself — `manifest.json`
   must be at the root of the zip). From the repo root:
   ```
   cd dist/chrome
   zip -r ../../tablocker-chrome-<version>.zip .
   cd ../..
   ```

**Submission flow (Developer Dashboard):**

1. From the dashboard, choose "New item" (for the first submission) or select the
   existing TabLocker item and "Package" → "Upload new package" (for updates).
2. Upload the zip built above. The dashboard will parse `manifest.json` and flag any
   manifest-level errors immediately.
3. Fill in the store listing:
   - **Icons**: a 128x128 icon is required for the store listing itself (separate from the
     in-package icons referenced by the manifest).
   - **Screenshots**: at least one screenshot (1280x800 or 640x400) showing the extension
     in use — e.g. the popup locking a tab, and the history view.
   - **Description**: short summary plus a detailed description of what the extension does
     (locks a tab to a home page and tracks that tab's history from that point) and its
     permissions (`storage`, `tabs`) with a plain-language justification for each, since
     Chrome's review process specifically checks that requested permissions are justified
     by the listing text.
   - **Privacy practices**: declare data collection/use per Chrome Web Store's privacy
     questionnaire, and provide a privacy policy URL if any user data is collected or
     transmitted.
   - **Category** and **language**.
4. Submit for review.

**Review time expectations:** typically ranges from a few hours to a few business days for
straightforward extensions with narrow permissions; can take longer if the review flags
the `tabs` permission for manual scrutiny. First-time submissions from a new developer
account sometimes take longer than subsequent updates.

**Publishing:** once approved, the item can be published immediately or held as a draft.
Publishing can be set to "Public" or restricted (e.g. "Unlisted" / "Private" to specific
testers) from the same dashboard listing page. Subsequent updates follow the same
upload-a-new-package flow and go through review again, though updates to already-approved
extensions are often reviewed faster than the initial submission.

## 3. Edge Add-ons

**Account setup (one-time):**

1. Sign in to the [Microsoft Partner Center](https://partner.microsoft.com/dashboard) with
   a Microsoft account and register/verify as an extension developer. Unlike Chrome, there
   is currently no developer registration fee for Edge Add-ons.

**Packaging:** the same `dist/chrome` zip built for the Chrome Web Store generally works
unmodified for Edge, since Edge is Chromium-based and consumes the same Manifest V3
format and the same base `manifest.json` (no `manifest.edge.json` overlay exists in this
repo, and none should be needed unless Edge-specific behavior diverges in the future).

**Submission flow (Partner Center):**

1. Create a new extension entry (first submission) or select the existing TabLocker
   listing and start a new submission (updates).
2. Upload the same zip used for Chrome.
3. Fill in the listing — mostly mirrors the Chrome Web Store listing (description, icons,
   screenshots, category, privacy policy URL), but expect minor differences in field names,
   image size requirements, and an additional "age ratings" style questionnaire that Chrome
   does not have. Reuse the same description/screenshots prepared for Chrome as a starting
   point rather than writing new listing copy from scratch.
4. Submit for review (Partner Center also runs its own automated policy/manifest checks
   before it accepts the submission for human review).

**Review time expectations:** commonly on the order of a few business days; Microsoft's
review has historically been slower than Chrome's for first submissions but is broadly
comparable for routine updates. Treat both as "budget several business days, don't assume
same-day approval" for planning purposes.

## 4. Firefox Add-ons (AMO)

**Account setup (one-time):**

1. Create/sign in to a [Firefox Account](https://accounts.firefox.com/) and register as an
   add-on developer at [addons.mozilla.org/developers](https://addons.mozilla.org/developers/).
   Registration is free.

**Packaging:**

1. Run `yarn build` to produce a fresh `dist/firefox`.
2. Run `yarn lint:ext` (see the Pre-submission checklist above) and fix anything it flags
   before packaging — this mirrors AMO's own automated validator closely enough that
   catching issues locally first saves review round-trips.
3. Zip the _contents_ of `dist/firefox` (root of the zip must contain `manifest.json`):
   ```
   cd dist/firefox
   zip -r ../../tablocker-firefox-<version>.zip .
   cd ../..
   ```

**Submission flow (AMO developer hub):**

1. From the [Developer Hub](https://addons.mozilla.org/developers/), choose to submit a
   new add-on (first submission) or upload a new version to the existing TabLocker listing
   (updates).
2. Choose distribution: "On this site" (listed publicly on AMO, subject to Mozilla's full
   review) versus "On your own" (self-distributed/unlisted, still requires Mozilla signing
   but skips public listing review). For a public extension, use "On this site."
3. Upload the zip. AMO's automated validator runs immediately and will reject the upload
   outright for manifest errors, disallowed API usage, or minification/obfuscation it
   cannot statically analyze — this is the same class of check `web-ext lint` performs
   locally.
4. Provide listing details: name, summary, description, category, icons, screenshots,
   support email/URL, and a privacy policy statement (required given the `tabs`
   permission touches browsing data).
5. Submit for review.

**Review process and expectations:**

- Every version, including updates, passes through **automated validation** first
  (immediate) and then a **manual human review** before it goes live on AMO, since Mozilla
  requires human review for all publicly listed extensions regardless of permissions
  requested.
- Firefox's manual review has a reputation for being **stricter and slower** than Chrome's
  or Edge's — reviewers read source more closely (including checking that the published
  package's behavior matches its source, which is part of why keeping the build
  reproducible and unminified/readable matters) and are more likely to ask clarifying
  questions or request changes before approval. Budget more calendar time for the first
  submission than for Chrome/Edge, and expect possible back-and-forth via the review
  comments.
- **Signing**: Mozilla requires all extensions distributed for Firefox (including
  self-distributed/unlisted ones) to be **signed by Mozilla**. For "On this site"
  submissions, signing happens automatically as part of approval — there is no separate
  manual signing step to perform. For self-distributed builds, `web-ext sign` (or the AMO
  API) can be used to obtain a signed `.xpi` without public listing, but that path is not
  needed for a normal public AMO release.

## 5. Post-publish

- **Version/changelog updates**: after a store submission is approved and published,
  there is nothing further to update in the repo itself for that release — `version` in
  `package.json`/`manifest.json` and the changelog entries are already handled by the
  `release-please` PR that was merged before packaging (see the Pre-submission checklist).
  If a store listing's own "what's new" / release notes field is separate from the repo's
  changelog (all three stores have one), copy the relevant `CHANGELOG.md` entry into that
  field manually at submission time — none of these stores read the repository automatically.
- **release-please and store publishing are decoupled.** Merging a `release-please`
  release PR only: bumps `package.json`'s version, updates `CHANGELOG.md`, tags the commit,
  and creates a GitHub Release. It does **not** build, package, or upload anything to
  Chrome Web Store, Edge Add-ons, or AMO — none of that is wired into
  `.github/workflows/release-please.yml` today. Publishing to each store after a
  `release-please` release is a fully manual step: build, zip, and walk through sections
  2–4 above for each store, per release. If this ever needs to be automated, that would be
  a separate CI job (e.g. triggered on the GitHub Release created by release-please) using
  each store's publishing API (Chrome Web Store Publish API, Partner Center submission API,
  AMO's `web-ext sign`/submission API) — no such automation exists in this repo yet.
- Keep a record (e.g. in the GitHub Release notes or a follow-up comment on the closing
  issue) of which store versions were actually published and when, since the three stores
  can end up on different released versions if one submission is delayed or rejected while
  others go through.
