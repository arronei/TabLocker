# Safari Deployment Plan

Status: not started. Safari support is explicitly out of scope for the current build (see issue #1)
because no Mac or Xcode installation is available to the maintainer. This document is a
self-contained walkthrough to execute once that changes. It assumes none of the work below has
been started and that no Safari-specific code or Xcode project exists in this repo yet.

This plan targets Safari on macOS (Safari App Extension) and notes the additional steps needed if
Safari on iOS/iPadOS is also targeted.

## 1. Prerequisites

Before starting, confirm all of the following are in place:

- A Mac running a currently-supported version of macOS.
- A current version of Xcode installed from the Mac App Store, including the Xcode command-line
  tools (`xcode-select --install`).
- An active enrollment in the Apple Developer Program (currently $99/year). A free Apple ID can
  build and run locally for development/testing, but distribution through the App Store and
  notarization for direct distribution both require a paid Developer Program membership.
- Access to [App Store Connect](https://appstoreconnect.apple.com) under the same Apple Developer
  account/team that will own the app record.
- A built, working extension output from this repo. The converter works from `build/`-style
  output directories, so run `yarn build` first and point the converter at `dist/chrome`. Use
  `dist/chrome`, not `dist/firefox`: this repo's `dist/firefox` manifest is merged with
  `manifest.firefox.json`, which adds Firefox-only keys (e.g. `browser_specific_settings.gecko`)
  that don't apply to Safari and shouldn't be carried into the converter input. `dist/chrome`'s
  manifest is closer to Safari's Chromium-leaning WebExtensions support and avoids that problem.

## 2. Converting the extension

Apple ships a CLI tool, `safari-web-extension-converter`, as part of the Xcode command-line
tools. It takes an existing Chrome/Firefox-style extension directory (manifest.json plus
scripts/assets) and generates a full Xcode project that wraps it as a native Safari Web Extension.

High-level steps:

1. Build the extension normally from the repo root: `yarn build`. This produces `dist/chrome` and
   `dist/firefox`.
2. Run the converter against `dist/chrome` (see the prerequisites above for why `dist/chrome`
   rather than `dist/firefox` is the right input here):
   ```
   safari-web-extension-converter dist/chrome
   ```
3. Choose a destination directory for the generated Xcode project when prompted (a location
   outside `dist/`, since `dist/` is a build artifact that gets wiped and regenerated by
   `build.js`).
4. The tool will ask whether to open the generated project in Xcode automatically, whether to
   use App Extension or Safari Web Extension conventions, and whether the extension should be
   available on macOS only or also on iOS/iPadOS. Answer according to the target platforms
   decided on at the time.
5. The converter produces a wrapper macOS (and optionally iOS) app target plus a Safari Web
   Extension target, with the original extension's manifest, scripts, and static assets copied
   into the extension target's resources.

The converter is a one-time scaffolding step, not a persistent build integration — see section 7
for the maintenance implications of that.

## 3. Xcode project setup

Once the project is generated, configure it in Xcode before it can be signed, run, or archived:

- **Bundle identifiers**: Set a reverse-DNS bundle identifier for the container app (e.g.
  `io.github.arronei.tablocker`) and a matching child identifier for the extension target (Xcode
  typically suggests `<container-id>.Extension` or similar; keep the convention Xcode proposes
  unless there's a reason to deviate). These identifiers must be registered to the Developer
  Program team used for signing and must be unique across the App Store.
- **Signing & Capabilities**: For each target (container app and extension), open the "Signing &
  Capabilities" tab, select the Developer Program team, and enable "Automatically manage
  signing" unless the project has a specific need for manual provisioning profiles. Xcode will
  create the necessary App IDs and provisioning profiles against the selected team.
- **Deployment targets**: Set the minimum supported macOS version for the container app and
  extension target. If also targeting Safari on iOS/iPadOS, set the minimum iOS/iPadOS version on
  the corresponding targets as well. Pick minimums based on the oldest OS versions still under
  consideration for support — check current Safari/WebExtensions feature availability per OS
  version before finalizing, since some WebExtensions APIs were added in specific Safari/macOS
  releases.
- **App icons and metadata**: Xcode's generated project includes placeholder app icon slots and
  an `Info.plist` for the container app; these need real icons and accurate display name, version,
  and build number before archiving. Note that `public/icons` in this repo currently contains only
  a `.gitkeep` placeholder (it's copied into `dist/<browser>/icons` by `build.js`'s icon-copy
  step, but no actual icon assets exist there yet) — producing real app icon artwork is a
  prerequisite task in its own right, not something that can be reused from this repo today.

## 4. Manifest/API compatibility caveats

Safari's WebExtensions implementation does not have full 1:1 parity with Chrome's or Firefox's.
Before or during conversion, spot-check the extension's actual API usage against Safari's
supported WebExtensions API surface (Apple publishes supported/unsupported API lists in its
Safari Web Extensions documentation, and this should be re-checked at conversion time since
support has been expanding release over release).

Specific things to verify for this extension:

- This repo's `manifest.json` currently requests only the `storage` and `tabs` permissions, and
  the background script runs as an MV3 service worker (`background.service_worker` +
  `background.scripts` with `"type": "module"`). Safari's MV3 background service worker support
  and its treatment of the `scripts` fallback key should be verified against the current Safari
  version's documented behavior, since Safari's manifest V3 service-worker support lagged behind
  Chrome's for a period.
- `chrome.storage` and `chrome.tabs` are both part of Safari's supported WebExtensions surface,
  but exact method/event coverage (e.g. specific `tabs` events or `storage.session` support) can
  differ and should be checked method-by-method against whatever calls the codebase actually
  makes at conversion time, not assumed from Chrome/Firefox behavior.
- This repo already depends on `webextension-polyfill` for the `browser.*` promise-based API
  surface, which smooths over some Chrome/Firefox callback-vs-promise differences and is
  generally compatible with Safari's WebExtensions implementation. It does not guarantee that
  every underlying API called through it is implemented in Safari — the polyfill normalizes
  calling convention, not API availability.
- Re-run through the manifest keys emitted by `build.js`'s manifest merge (`manifest.json` plus
  any `manifest.<browser>.json` overlay) and confirm each key Safari's converter/Xcode project
  expects is present or has a Safari-appropriate equivalent (e.g. `browser_specific_settings` is
  Firefox-specific and is not expected to carry over into the Safari-generated manifest).

Any gaps found here should be resolved (feature-detected, polyfilled, or explicitly gated off for
Safari) before submitting for review, not discovered during App Review.

## 5. Testing

1. In Xcode, select the container app scheme and run it on the desired destination (a Mac, or a
   Safari-on-iOS simulator/device if targeting iOS/iPadOS). This installs the container app,
   which in turn registers the extension with Safari.
2. In Safari's settings, enable the Develop menu if not already enabled (Safari Settings >
   Advanced > "Show features for web developers", naming may vary by Safari version), then in the
   Develop menu enable "Allow Unsigned Extensions" so a locally built, non-notarized extension can
   be loaded for testing. This setting resets on relaunch/restart in some Safari versions, so it
   may need to be re-enabled each testing session.
3. Enable the extension itself in Safari's Extensions settings/preferences pane and grant it
   whatever site/tab permissions it requests.
4. Use Safari's Web Inspector (via the Develop menu, pointed at the extension's background
   content or popup) to view console output and debug the extension's scripts running inside
   Safari, similar to Chrome/Firefox devtools.
5. Exercise the extension's actual functionality (locking a tab, tracking that tab's history)
   manually inside Safari. This repo's existing Playwright e2e suite (`playwright.config.ts`,
   `e2e/`) currently only launches Chromium against the unpacked `dist/chrome` extension (see
   `e2e/fixtures.ts`) — it does not run against Firefox, Safari/WebKit, or any other browser today.
   Extending it to cover Safari/WebKit extension contexts is out of scope for this plan; Safari
   testing here is expected to be manual through Xcode/Safari's tooling rather than folded into
   the existing automated e2e run.

## 6. App Store Connect submission

Safari extensions on macOS ship as a full container app with an embedded extension — there is no
"extension-only" listing. That means the App Store review guidelines applicable to a normal macOS
app apply to the wrapper app, not just extension-specific guidelines.

1. In App Store Connect, create a new app record: pick the bundle ID registered in Xcode, set the
   app name, primary language, SKU, and platform (macOS, and iOS/iPadOS if applicable).
2. Fill in required metadata: app description, keywords, support URL, marketing URL (optional),
   privacy policy URL (required — this extension tracks tab browsing history locally, so the
   privacy policy needs to accurately describe what is and isn't collected/transmitted), and the
   app's primary category.
3. Provide required screenshots for each supported platform/device size App Store Connect asks
   for (macOS screenshots at minimum; additional iOS/iPadOS device-size screenshots if that
   platform is included). Screenshots should show the actual extension UI (popup, history view)
   since App Review checks that screenshots reflect real app functionality.
4. Complete the App Privacy questionnaire (data collection/use disclosures) — since this
   extension deals with browsing history, be precise about whether any data leaves the device
   (it currently doesn't, based on the `storage`-only persistence model, but this should be
   re-verified against whatever the codebase does at the time of submission).
5. Archive the container app in Xcode (Product > Archive) and upload the build through Xcode's
   organizer, or via `xcodebuild` / Transporter, to App Store Connect. Uploaded builds go through
   Apple's automated processing (which includes notarization for App Store builds automatically)
   before they're selectable for a version submission.
6. Submit for App Review. Typical App Review turnaround has historically been roughly 24-48 hours
   for the first review pass, but this varies and should not be treated as a hard SLA — build in
   schedule slack, especially for a first-time submission of a new app record, which sometimes
   takes longer or draws additional scrutiny (e.g. Guideline 2.3 accuracy checks, or extension
   permission justification requests).
7. **Notarization vs. App Store distribution**: builds uploaded through App Store Connect for App
   Store distribution are notarized as part of Apple's processing pipeline automatically — no
   separate manual notarization step is needed for that path. If instead distributing the app
   directly (outside the App Store, e.g. via direct download + Developer ID signing), a separate
   manual notarization step is required: archive with a Developer ID signing identity rather than
   an App Store distribution identity, then submit the build to Apple's notary service
   (`xcrun notarytool submit`) and staple the notarization ticket (`xcrun stapler staple`) before
   distributing it. Decide up front which distribution path this project wants — App Store only,
   direct-distribution only, or both — since that determines which signing identity/export method
   to use at archive time.

## 7. Ongoing maintenance note

The Xcode project generated by `safari-web-extension-converter` is a separate, standalone
project — it is not wired into this repo's `build.js` pipeline and does not stay in sync
automatically. Concretely:

- Every time source changes are made in `src/` that should ship to Safari, the wrapped extension
  resources inside the Xcode project need to be updated to match. There are two ways to do that:
  either re-run `safari-web-extension-converter` against a fresh `yarn build` output and let it
  regenerate/update the wrapping project, or manually copy the updated built files
  (`dist/chrome/**`) into the existing Xcode project's extension resources folder. Re-running the
  converter is generally safer for staying current with Apple's expected project structure, but
  may require re-applying any manual Xcode configuration (signing, bundle IDs, deployment
  targets, icons) if it doesn't preserve them across a re-conversion.
- This means Safari support has a real, recurring cost: it is not "free" once shipped once. Any
  process for cutting a new extension release should explicitly include a Safari sync step (or a
  documented decision to let Safari lag behind and batch updates), rather than assuming Safari
  automatically tracks the Chrome/Firefox builds produced by `build.js`.
- Because of this cost, it is worth deciding at pickup time whether Safari support is maintained
  on every release, on a slower cadence, or only revisited on demand — and documenting that
  decision so it doesn't have to be re-derived each time this plan is picked back up.
